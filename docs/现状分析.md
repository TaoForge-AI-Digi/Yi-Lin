## 一、已定架构决策

设计总纲是早期蓝图，以下已由当前实现定型，后续以当前为准：

| 维度 | 决策 |
|------|------|
| 项目结构 | Monorepo: `apps/client/` (Vue 前端) + `apps/server/` (Node 后端) |
| 交互方式 | WebSocket + Vue 前端，非 CLI |
| 技术栈 | TypeScript / Node.js (Hono) + SQLite |
| 角色注册 | 当前: JSON 文件 (`data/characters.json`)。目标: 每个角色独立目录 (`data/characters/{id}/`)，含 `character.json` + `soul.md` + `user.md` + `memory.md`，便于导入导出和分享 |

### 仍待对齐的差异

| 维度 | 设计总纲期望 | 当前实际 | 差距等级 |
|------|-------------|---------|---------|
| 权限模型 | 两阶段：L1约束 + L2全局策略 | per-tool 三级权限 (allow/ask/deny) | 待对齐 → 以设计总纲为准 |
| 角色术语 | `role` = main/sub/both | `mode` = primary/subagent/all | 待对齐 → 以设计总纲为准 |

## 四、当前代码与设计总纲的冲突

### 4.1 CharacterRecord 缺少字段

统一使用 `role`，前后端口径一致，当前 `mode` 字段废弃。

```typescript
// 前后端统一
interface CharacterRecord {
  // 存在:
  id, name, description, avatar, color, memory, model, provider,
  tools, permissions, maxSteps, enabled, builtIn
  // 替换 mode:
  role          // 'main' | 'sub' | 'both'    ← 前后端统一口径
  // 新增:
  groups        // string[]                   ← 角色所属组列表（多对多）
  soul_path     // string                     ← Soul 文件路径
  memory_path   // string                     ← Memory 文件路径
  user_memory_path // string?                 ← User 文件路径（可选）
  skill_bindings   // string[]                ← 绑定到角色的技能列表
  default_strategy // 'Plan' | 'Ask' | 'Bypass' ← 默认策略
}
```

### 4.2 无工具级约束

设计文档 `ToolBinding` 支持参数级约束：
```yaml
constraints:
  allowed_paths: ["/project/src/**"]
  denied_paths: ["/project/secrets/**"]
  max_file_size: "1MB"
  allowed_commands: ["npm", "go", "git"]
  denied_patterns: ["rm -rf", "sudo"]
```

当前只有 workspace 级别的路径逃逸检查，无细粒度约束。

### 4.3 Skill 不与角色绑定

设计文档：技能绑定在 `CharacterRegistration.bindings.skills` 列表中，每个角色有不同的技能库。

当前：技能是全局的，`/api/skills` 返回所有技能，`CharacterRecord` 无 skills 字段。

### 4.4 loop.ts 单层循环

设计文档：外层循环（会话管理 + Sub-agent 信号 + 上下文压缩 + 顿悟）与内层循环（LLM 推理 + 工具调用）分离。

当前：`runCharacter()` (`apps/server/src/agent/loop.ts`) 一个函数包含全部逻辑。

### 4.5 权限模型以设计总纲为准

设计总纲的**两阶段校验**（L1 身份绑定约束 + L2 全局策略拦截）替代当前 per-tool 三级权限。

| 当前 (将被替换) | 目标 (设计总纲) |
|----------------|----------------|
| `permissions: { edit: 'ask', bash: 'deny', webfetch: 'allow' }` | L1: `ToolBinding.constraints` 参数级约束 |
| `checkPermission()` 返回 allow/ask/deny | L2: `SessionState.current_strategy` = Plan/Ask/Bypass |
| category 映射 (edit/bash/webfetch) | `DANGEROUS_TOOLS` 硬编码列表 |
| 无工具级约束 | allowed_paths, denied_paths, max_file_size, allowed_commands 等 |

权限模型以设计总纲为准，因此策略引擎和约束校验优先级提升。按改动量从低到高：

1. **加 `groups` 字段到 `CharacterRecord`** — 数组字段 + 组管理界面 + 聊天弹窗按组筛选 + 角色编辑侧边栏按组分栏
2. **实现 SessionState + 策略命令** — 新增 `strategy` 路由 + WebSocket 命令，改动 `loop.ts` 入口
3. **拆分 `loop.ts` 为 outer/inner** — 重构不改变行为，为后续功能铺路
4. **实现 `delegate_task` + Sub-agent** — 核心分形能力
5. **实现 `skill_load` 工具** — 让 LLM 能从对话中加载 skill
6. **Skill 绑定到角色** — 给 `CharacterRecord` 加 `skills` 字段
